この文書の位置づけ 設計ディスカッション記録であり正規仕様ではない。実装は未着手（build は engine cutover 後・home 確定後）。 記載の記法・API は叩き台。明日の demo 用の最小スクリプトは gauge-by-progress の脇 spike（本来の engine 開発を邪魔しない範囲）。

0. 背景

三輪氏（音楽家）の「譜面表示できる？」が発端。相手が音楽家なので 本物の五線譜が要る（grid/piano-roll では不可）。 対象は Pitch DSL のみ（audio sequence は対象外）。

1. 核心課題 = MLTS（Multi-Layered Temporal Structure）

なぜ既製ツールで書けないか MLTS = 層（sequence）ごとに beat/tempo が独立し、小節線が非整列にずれ込む（polymeter／polytempo）。 現代西洋記譜は共有小節線（全段が同じ barline グリッド）を大前提とするため、VexFlow 素・Verovio・MusicXML・OSMD・さらに LilyPond でも native に書けない領域。 ＝ レンダラは自作が必須。問いは「何の上に作るか」だけ。

正本: Pitch DSL Spec v1.1 §9.4「ポリメーター/ハーモニックリズムの干渉（MLTS 拡張）」/ BEAT_METER_SPECIFICATION.md （1小節 = (60000/tempo)×(numerator/denominator×4) ms・per-seq 独立 → 小節線がドリフトし LCM で再同期）。

2.1 なぜ VexFlow か（MLTS が決め手）

  • VexFlow は低レベル programmatic ＝ 層ごとの小節線を自前 x 座標で置ける。これが MLTS（ずれる小節線）に必須。小節内の難所（音符グリフ・臨時記号・連桁・spacing）は VexFlow に任せ、自作するのは 「MLTS レイアウト層（層ごと bar grid + 時間軸での層間整列）」だけ。MIT・active(v5)・SVG+CSS でアニメ可。
  • OSMD は不採用: 価値 = MusicXML から標準記譜を自動レイアウト。だがその自動化は整列小節線を前提＝MLTS が破る部分そのもの。MusicXML は MLTS を表現できず、生成も冗長。OSMD は VexFlow の上なので MLTS には VexFlow の低レベル制御が結局必要で、OSMD 層はむしろ邪魔。
  • Verovio 不採用: engraving は最高だが LGPLv3 + MEI/MusicXML（整列前提）+ WASM 重 → MLTS では優位消失。
  • publication = LilyPond だが MLTS は LilyPond でも frontier（ずれる小節線は native 不可）。→ MLTS の出版答えは「拡張 VexFlow / 自作レンダラ」。LilyPond は標準還元できる部分の補助。
長期: 拡張 VexFlow が MLTS 記譜システムの正攻法 MLTS はどの既製ツールも native に書けない ＝ どの道レンダラは自作する。VexFlow を MLTS-aware に拡張すれば live と publication を1レンダラに統一できる（live は今・出版品質へ後で磨く）。 notation-model（中間表現）→ 複数レンダラ（VexFlow / 将来 LilyPond）も将来余地。

3. real-time = 自前（live-coding 整合）

engine が timing を駆動し、VexFlow は「描画 + カーソル overlay」だけ。カーソルは VexFlow/lib 自身の synth でなく OrbitScore の transport（getTransportPosition() / now_sec）から x 座標を出して駆動する。 組込 playback/cursor の価値は下がり、lib は engraving + programmatic 動的描画 + license + メンテで選ぶ ＝ VexFlow が最クリーン。 VexFlow は SVG + CSS アニメ・要素グループ化・音符の動的追加に対応（参考: mrk21.hatenablog.com/entry/2018/06/04/103857）。

4. データブリッジ（engine 非依存・interpreter 層）

scout（#339 関連調査）で確認。すべて interpreter レベル ＝ engine 非依存（今の SC エンジンでも動く）:

  • 音符 + timing + pitch: interpreter.getState() の per-seq timedEvents（TimedEvent.pitch = degree/alteration/oct/detune・public）。
  • staff 位置: resolveDegree（packages/engine/src/midi/index.ts public）で degree→MIDI note。
  • per-layer MLTS grid: per-seq beat / tempo / length（Sequence.getState()）→ 1小節長。
  • playhead / 次 swap: global.getTransportPosition() / getQuantizedEffectPosition()。
  • pitched 判別: isMidi()。
  • 既存 headless 実行: npm run midi-run -- <file.orbs>（degree→MIDI 解決の既存パス・抽出に再利用可）。
  • 最小 tap: getState() polling + 小さな WS（core 改変ゼロ）。clean 版は InterpreterV2.addObserver() 追加（~10行）。WS サーバは net-new（~50行）。

→ 譜面 view は engine 非依存の read-only consumerとして成立。audio path / engine core / .vsix を触らない。

5. home（後決め・優先は engine cutover）

  • 2.0.0 .vsix には載せない（owner 判断・筋が悪い）。

6. 研究新規性

MLTS 記譜には標準が無い → その視覚言語を設計すること自体が貢献（論文の芽。owner の研究志向と噛む）。

締切は外した（間に合わなくて良い）。最小 = pitch DSL を VexFlow で描画するスクリプトを用意するだけ（static でも可・「なんらか見せれれば良い」）。 scratchpad で・本来の engine 開発を一切邪魔しない範囲で進め、進捗で判断。良ければ後で正規 spike へ昇格、ダメなら捨てる。 実装案: midi-run 経路 or getState() + resolveDegree で pitch .orbs → 音符抽出 → VexFlow で per-layer offset barline 付き静的描画（MLTS の見せ場）。
