1. メタ原則 — 業界規格採用は leverage

OrbitScore が業界規格（VST3 / CLAP / AU プラグイン形式・LinkAudio・MIDI・他 DAW へのオーディオ）を取り入れるのは、 単なる互換性ではなく leverage（てこ） として。理由は 2 つ:

  1. 手段の組み合わせを柔軟にできる — ユーザーも私たちもツールを混在させられる（interop > 囲い込み）。
  2. 私たちが先に実装しなければならないものにリソースを集中する — 規格に乗れる所は乗り、 自分たちにしか作れない fundamental（audio DSL / scheduling / engine の差別化要素）に希少な開発リソースを寄せる。

以降の §2–§7 はすべてこのメタ原則のインスタンスである。

2. Engine は fundamental に保つ — effects/instruments は plugin

North star（plugin 作業全体の指針） fundamental engine（audio DSL + scheduling + plugin host substrate + DSL→plugin 制御）に effects/instruments の DSP を抱え込まない。1st-party を作るときも engine 内ではなく別の CLAP/VST plugin として作る（または 3rd-party で代用）。 VST3/CLAP/AU 対応は、その「engine 外 plugin で effects/instruments を成立させる」delivery 機構であり、 format breadth が目的ではない。

なぜこれが正しい資源配分か:

  • 1st-party effect を別 plugin にすれば OrbitScore 以外の CLAP/VST host でも再利用できる。
  • engine と独立に開発でき、engine repo / cutover scope を汚さない。
  • 3rd-party plugin は 無償のレバレッジ。
  • 結果、engine の開発リソースは fundamental（host + DSL）に集中できる。

帰結: engine に compressor/limiter/normalizer/EQ 等の DSP を実装する提案はしない。必要なら「別 CLAP/VST plugin」か「hosted 3rd-party」で解く。

3. Format 非依存 substrate（γ の OOP plugin host）

γ の out-of-process plugin host は format 非依存の substrate。plugin format = どの child binary を spawn するかだけで、各 SDK は child プロセス内に隔離される（fault 隔離の鏡）。

substrateconsumers（child binary・SDK は child 内隔離）
M1 effect transport（file-backed mmap・DONE #360）CLAP effect child ✓ ／ VST3 effect child（追加）／ AU effect child（追加）
M2 instrument IPC（per-block event/param・次）CLAP instrument child ／ VST3 instrument child（追加）／ AU instrument child（追加）
唯一の plan-affecting 決定 — M2 IPC を format-neutral に M2 の event/param IPC は format-neutral（note + param/CC の意味論）で仕様化する。CLAP イベント形に寄せない。 PluginEvent::NoteOn{key,channel,velocity} は既に neutral。この規律だけが 「VST3/AU = 後から additive」と「後から作り直し」を分ける。さらに param/CC change を neutral に運ぶことが、 「DSL から plugin を CC 制御」（§7 EQ-from-DSL）の基盤になる。

4. Cutover parity bar の精緻化

cutover #108 の gate は 現行 .orbs DSL が SuperCollider と同等に鳴ること（A3 + A4 + γ effects 機構）であり、 plugin format の広さとは直交する。NEXT_STEPS §2 が挙げた parity gap のうち master effects を実情に合わせて緩める:

master compressor / limiter / normalizer は cutover bar から外す defer これらの DSL verb（global.compressor()/limiter()/normalizer()）は scsynth 時代も実際には未使用だった （effects は外部 = Ableton 等に委譲）。よって SC parity は実質的な gap ではない → 現状の warn + no-op を据え置き、cutover を更に lean にする。
根拠は「未使用 + effects は engine の外（plugin / 外部 DAW）に置く」原則であって、「LinkAudio が effects を担うから」ではない（§5）。将来 effects が要れば hosted plugin で。

refined cutover gate = A3 ✓（slice varispeed）+ A4 ✓（LinkAudio）+ parity-gap followup（hard-stop-all / Gap5 / Finding2）+ capture seam（耳なし実時間検証）。master 自前 effects は load-bearing ではない。

5. LinkAudio は出力の 1 つ（銀の弾丸ではない）

framing 訂正 LinkAudio を「effects の経路」と中心に据えるのは誤り。OrbitStudio / OrbitEngine から見て LinkAudio は多数ある出力オプションの 1 つにすぎない。 必要なら使う・不要なら使わない。
  • OrbitStudio に per-track recording があれば LinkAudio を使わない構成もあり得る。
  • 他 DAW へオーディオを流し込んで録音、でもよい。
  • MIDI 出力（pitch DSL v1.1）も並ぶ出力経路の 1 つ。

出力/ルーティングは fundamental engine の外側の柔軟な層（メタ原則 §1）。どれを使うかはユーザー/構成次第。

6. OrbitStudio のスコープ

  • 差別化に効く所は self-contained を目指す（「なるべく OrbitStudio で完結」）。
  • ただし 完結 ≠ 全部 1st-party。録音・effects 等は interop で解いてよい（他 DAW へ録る・plugin を host する）。
  • per-track recording は nice-to-have（あると扱いやすい・録り忘れ防止）だが 1st-party 必須ではない（余裕があれば実装）。

7. δ — VST3 / AU（additive・優先度順）

「VST3 は cutover の前か後か」= 問いの軸が違う

cutover の gate は現行 DSL の SC parity（§4）で、plugin breadth と直交。よって VST3 は cutover を塞がず・待たず、§3 の format 非依存 substrate に乗る additive トラックとして 優先度（依存ではない）で並べる。 実際の次の関門は M2（CLAP instrument にも必要 = 既にプラン内）。VST3 は乗るだけでロードマップは並び替わらない。

  • VST3 effects = M1 substrate に VST3 effect child を追加（低コスト・実 plugin 資産でアーキを検証できる test 価値が高い）。
  • VST3 instruments = M2 substrate に VST3 instrument child を追加（pitch DSL と結節するため重要）。
  • AU は急がない（owner）。
gate — Step0 spike + ライセンス一次確認 VST3 を child 内で host する部分は net-new・未実証の Rust 作業（自社 doc 評価「最難」）。 γ が spike（#349/#351）で確信を得たのと同様、δ scope 確定の前に VST3 Step0 spike （実 VST3 を 1 つ load → vst3 crate で sample-exact な 1 block を出す）を通す。 併せて VST3 SDK の現行ライセンスを一次確認（「SDK 3.8 = MIT」は自記述で primary 未確認。 permissive graph に置けるか否かを左右する。OOP child のプロセス境界は license に関係なく fault 隔離は効く）。

8. EQ-from-DSL post-cutover want

owner の本命は DSL から操作できる EQ（comp/limiter と違い音響表現に直結する creative effect）。 ただし急がない — VST/CLAP を CC で制御すれば済むため。 これは §3 の M2 format-neutral param/CC path の直接の消費者であり、 新機構なしに「hosted CLAP/VST3 EQ + M2 param CC」で成る（post-cutover）。
