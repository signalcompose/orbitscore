# Post-2.0 Pitch モデル再設計 覚書（root / key / scale / 転調）

> **ステータス: 探索段階・未確定・post-2.0。** 2026-06-19 に大和さんと議論して解けた pitch モデルの記録。
> **注意**: ここには v1.1/Phase 2 の **Known Decision を将来変更する候補**（`.root()` 廃止、key への mode 付与など）が含まれる。`docs/specs-v2/DESIGN_DISCUSSION_RECORD.md` の確定決定 #1-32 を**今書き換えるものではない**。採用時は spec 先行で更新する。
> 前提: **2.0.0-dev を先に確定**してから着手（[[POST_2.0_ROADMAP_NOTES]] の deferred 方針）。これは engine 非依存（MIDI/pitch）なので engine 載せ替えとは独立に進められる。

## 1. 解けた全体像 — 2軸 × カスケード

pitch は**直交する2軸**で決まり、両軸とも **global → sequence → group** のスコープを下る:

| 軸 | global | sequence | group（最具体） |
|---|---|---|---|
| **anchor（root/tonic）** | `global.key("C")` | `seq.key("F")`（マルチキー/ポリキー） | `(...)F`（絶対）/ `(...)I`（相対） |
| **lattice（scale/mode）** | `key(...).mode(...)`（省略=Ionian） | （seq 既定） | `(...).mode(...)` |

`key` は**両カスケードの頂点 = (tonic, デフォルトラティス)**。これが「key だけ tonic で scale が宙ぶらりん」だった据わりの悪さの解。

## 2. root（グループ後置に一本化）

- **`.root()` メソッドと `seq.root()` は廃止**。root はグループ後置記法のみ。
  - **絶対 = 音名**: `(...)F` `(...)Bb`（`A–G` + accidental）。key 不要。
  - **相対 = 大文字ローマ**: `(...)I`〜`(...)VII`、+ **前置 accidental**（`(...)bII`、`(...)##V`）。key に対して解決。
- **大文字ローマのみ**（小文字ローマ=マイナー等のクラシック記号は不採用）。**quality は `(...)` の中身が単独で決める**ので、ローマは「どの度数がルートか」だけ。
- **字句的に衝突しない**: 絶対(A–G) と 相対ローマ(I/V始まり、または b/# 始まり) は先頭文字集合が重ならない。残る曖昧性は「その後置スロットの同名変数」だけ → そのスロットで A–G / I–VII を pitch リテラル予約すれば解消（前回案の `(...)F` 単独曖昧性より遥かに軽い）。
- **綴り保持**: `##V` を `VI` に正規化しない（既存 §7-0 reversibility = `D# vs Eb` と同じ思想）。
- **#280 はこの再設計で消滅**（`seq.root(note-name)` 拒否バグは `seq.root` 廃止で無くなる。anchor は `seq.key`）。

## 3. key / lattice（mode）

- `key("C").mode("minor" | customVar)`、**省略で Ionian**。
- **lattice エンジンは実装済み**: 任意長ラティス + 明示 `modePeriod`（degree-resolution.ts:73、コメントで「7音前提を捨てる」明記）。→ **ホールズワースの2オクターブスケール等（長ラティス + period=24）は表現可能**。custom は `var x = mode(1,2,b3,…)`（§2.2）。
- **net-new（コア数学ではなく配線/砂糖）**:
  1. global/key 階層のデフォルトラティス setter（現状 `.mode()` はグループ専用、seq では reserved）。
  2. 組み込み名前付き mode（`"minor"`/`"aeolian"`/`"dorian"`/`"harmonic minor"`…）= alias テーブル。
  3. `key(...).mode(...)` チェーンのパース。
  4. 後置 root のパース（note vs roman）。
- **要注意**: period ≠ 12 のスケールで **`^N`（オクターブシフト）の定義** — `^1` は +12 リテラルか +period か。voicing とも絡む。

## 4. コード名は別レイヤー（クロマティック）

- `m7` / `Cmaj7` 等の**和音シンボルは自前の固定インターバル（クロマティック）**で、key/mode のラティスに**曲げられない**。
- → 「bare 度数 `[1 3 5]` = スケール追従 / コード名 = 絶対 quality」という**意図的な2層**（旧「二重解決」の正体。バグではない）。

## 5. 書き味（1モデルからの idiom）

- **相対で全書き**（モーダル曲）: `key.mode` で縛り `(...)IV` 等。key 一行で全体移調できる（transpose 不要）。
- **絶対で局所上書き**: 途中 `(...)F`（借用・アウト）。
- どちらも、両方混在も、同じモデルから出る。

## 6. 移調 / モード変更 / 転調 は別操作（混同しない）

- **移調 (transpose)** = 全ピッチを N 半音ずらす最終段オフセット。絶対 root も動く＝表記そのままで移調。名前は **`transpose()`**。
  - ⚠️ **`transport` は使わない**（Tracktion `TransportControl` = 再生ヘッド/クロックと衝突）。
- **モード変更** = ラティス差し替え（`key.mode(...)`）。
- **転調 (modulation)** = key 自体が時間で変わる → **ソング層（§7）**。

## 7. ソング/アレンジ層（別構築物・post-2.0・"使って検証" 前提）

> 2026-06-19 に設計を詰めた層。**記法・名前は暫定**。大和さん曰く「ここからは使ってみないと分からない」→ 2.0.0-dev 確定後、実使用で検証してから固める。転調（key over time）の置き場所であり、複数 seq を時間軸に構成する層。音符/グループ層（§1-6）は「key の中の相対/絶対」だけ持ち、**ライブコーディングはそれで十分**（確認済）。

### 7.1 playable 統一
`RUN`/`LOOP` は **seq でも song でも**受ける。song は seq を含み、**song を song に入れ子**にできる（再帰合成: verse/chorus → 曲 → セット）。「再生できるもの = playable（seq | song）」という一段上の抽象。

### 7.2 part = co-active な seq の集合（縦軸は seq 固有・連想配列キーは不採用）
- part は「同時に鳴る seq の集合」。**どのトラックか = 各 seq 固有**（名前 + `.output` チャンネル）。song 側でトラックキーを振り直さない（連想配列キーは seq の正体と**二重**になるので不採用）。
- part の各要素は **seq（節を通して鳴る）** か **サブタイムラインのリスト `[...]`（節内の出入り。`_`=休み）**:
  ```
  verse = part([
    melA,
    walkA,
    [ _*2, beat*6 ],   // 節内アレンジ: 2小節休んで6小節
  ])
  ```
- これは `[...]`・`*n`・ネストという既存の時間構造を scale 違いで再帰適用しただけ（フラクタル）。

### 7.3 MLTS の核心 = 節の「基準クロック」を1つ選ぶ
メーター違い（5/4 と 3/4 等）では "小節" が共有単位でないので、節尺は基準を決めないと定義不能（「8 of 何?」）。候補:
- **(A) global グリッド（既定）** — 既存の loop quantize 基準。
- **(B) conductor（指揮者）** — 節内の1トラックを基準に指名（§7.4）。
- **(C) ポリメーター共通周期(LCM)** — 全員が小節頭で揃う点を境界に（途中切り無し）。
- **(E) wall-clock** — ポリテンポ（テンポ自体が違う）ならこれしか成立しない。

### 7.4 conductor（指揮者）
- 節内の基準トラックを局所指名。比喩通り「誰の小節が "その小節" か」を決める。
- **インライン標識 `conduct(...)` が第一**（数値 index 参照は並べ替えで壊れ脆い）。seq は普通変数化するので無名トラックは稀だが、インライン定義したトラックも `conduct(...)` で包めば指せる。名前参照 `.conduct(seqName)` も可（seq の既存正体を使う＝二重でない）。
- **指揮者の書かれた長さ = 節の長さ**。非指揮者はその span をループで埋める。→ **part レベルの `*N` は不要**（`*N`/順次/key は song 側へ）。「指揮者は書き切る／伴奏はループで潜る」= アレンジの実際。
- 指揮者未指定なら (A) global グリッドが既定。`conduct` はそれを局所上書きするオプトイン。

### 7.5 境界ポリシー（指揮者と直交するノブ）
「基準クロック（何を数えるか）」とは別に「非指揮者を境界でどう扱うか」が直交:
- **hard（既定）**: 境界で非指揮者を途中でも切る（既存 quantize と地続き）。
- **musical**: 非指揮者が自分の小節頭に来るまで待つ（途中切り無し・(C) 寄り）。

### 7.6 song / フォーム層（順次 + 尺 + key）
`*N`（節の反復）・順次（A の次に B）・per-part key（転調）は song 側:
```
song([ verse*2, chorus.key("Eb"), verse, chorus.key("Eb") ])
```
- **順次合成（sequential）が新しい軸**。simultaneous は既存（juxtaposition / 集合）。**#254 section 変数（カンマ区切り multi-bar）が順次の既存の種**で、song はその延長（ゼロ新規ではない）。

### 7.7 命名
- `song`（名詞=出来上がり）。動詞が要るなら `arrange`。`exec` は shell 連想で却下。**型と実体を同名にしない**（`var verse = part(...)`, `var tune = song(...)`）。
- 移調は `transpose()`（**`transport` は Tracktion `TransportControl`＝再生ヘッドと衝突するので不可**）。

### 7.8 位置づけ
live-coding core より後の**アレンジ/タイムライン層**。MIDI/pitch 寄りで engine 非依存だが、規模が大きく **"使って検証" が前提**。

## 8. Open Questions

**pitch（§1-6）**
- `^N`（オクターブ）と `modePeriod` の関係（マルチオクターブスケール）。
- 組み込み名前付き mode の一覧と、minor の既定（natural/Aeolian）。
- `.root()` 廃止に伴う既存 example/ドキュメントの移行。

**song 層（§7・"使って検証" で詰める）**
- 境界ポリシー hard/musical の既定と per-section オプトの記法。
- 節境界をまたぐ seq の継続 vs リスタート（同一 seq インスタンスは継ぎ目なく続くか。例: drums が verse→chorus で連続）。
- ポリテンポ（テンポ違い）時の基準クロック（wall-clock / leader 指定）。
- 順次合成（sequential）と #254 section 変数の統合度（song は #254 の延長で組めるか）。
- 名前・記法の最終確定（`song`/`part`/`conduct`/`arrange`）は実使用後。
