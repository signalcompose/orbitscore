2. 背骨: cutover #108 までの parity gap

engine-first の到達目標は cutover #108（Rust を default backend にし scsynth を退役）。 そのためには現行 .orbs audio DSL が opt-in で SuperCollider と同等に鳴る必要がある。 #304 が「見かけの parity を作らない」ために warn / no-op に倒した3点が、そのまま cutover の parity gap になる。

#304 で倒した挙動現状（opt-in 時）埋めるフェーズ
time-stretch（slice rate≠1.0 / fixpitch() / time()）1回 warn して自然尺（rate=1.0）で発音A3（Signalsmith・#213 / #239）
outputChannel（LinkAudio）1回 warn してhardware バスで発音A4（Rust 隔離 GPL + lock-free）
master effects（compressor / limiter / normalizer）1回 warn してno-opγ（effects = plugin = out-of-process sandbox）
含意 cutover の前提は A3 + A4 + γ（effects）。このうち A3 / A4 は in-process / 隔離モジュールで完結し γ（CLAP plugin host）に依存しない＝近接で着手可能。effects parity だけが γ（最大スコープ）を必要とする。 daemon 側 hard-stop-all（下記フォローアップ）も、長サンプルでの cutover 品質には効く。

3. この後の実装計画（engine-first 残フェーズ）

アーキ §2.1–2.4（楽器=in-process / effects+3rd-party=sandbox / audio DSL⊇pitch / egress は音だけ bridge）は不変。順序はおおむね master plan の通りだが、依存関係を接地し直して下に整理する。

A3 — time-stretch contained

目的: slice rate≠1.0 と fixpitch()/time()（現 stub・#213）を実機で鳴らす。依存: in-process・γ 不要。S2/#304 の RustEnginePlayer 基盤の上。license: Signalsmith（permissive）必須・Rubber Band(GPL) 回避。Done: rate≠1.0 の slice が尺合わせで鳴る + 既存全緑。規模: 中。open: Signalsmith の Rust binding（ssstretch/signalsmith-stretch）の調達・成熟度は未確認（要 spike）。委譲: DSP crate は Sonnet 可、DSL seam は Opus。

A4 — LinkAudio（Rust 隔離 GPL）contained

目的: outputChannel（LinkAudio・#209）を Rust 経路で鳴らす + Link tempo lead（#283）。依存: 隔離モジュールで完結・γ 不要。license: Ableton Link は GPL → 別プロセス/別 crate の隔離モジュール（unique 不変条件・freemium の土台）。Done: LinkAudio channel が Rust 経路で鳴る + lock-free 化。規模: 中〜大（GPL 隔離境界 + lock-free が要点）。委譲: 隔離モジュールは Sonnet 可、境界設計は Opus。

daemon CLAP 統合 — γ の前提 大

目的: S1/S1b の in-process clack-host を production daemon に統合する（現状 standalone spike・S2 はサンプル経路のみ）。依存: γ（effects/3rd-party）の前提フェーズ。carry-forward（A0 §13・bot 検出）をここで踏む: ① teardown は audio thread で stop_processing()（暗黙 Drop 依存は strict プラグインで UB）② request_callback を lock-free 通知に ③ EventBuffer を固定サイズ CLAP event ring に。規模: 大。委譲: Opus 直列（RT/CLAP correctness）。

γ — out-of-process sandbox（fault ②）最大・最高リスク

目的: effects + 3rd-party plugin をプロセス境界で隔離し、daemon が 3rd-party crash を生存（C-ABI segfault はプロセス境界でのみ封じ込め）。これで master effects parity も埋まる。依存: 上の daemon CLAP 統合。新規: shared-mem audio / RT-safe event IPC / watchdog（本リポジトリに前例なし＝別スパイク）。fault 三層との関係: ② を作る。① recovery floor（#300）は前提として既に在る。規模: 最大・最高リスク。委譲: Opus 直列（spike）。

δ — 3rd-party VST3 / AU 最後

目的: 世の VST3/AU をホスト。成熟度順 CLAP > AU > VST3。依存: γ sandbox。規模: 最難・最後。VST3 は SDK 3.8 で MIT 単独化済（copyleft 障壁は消滅）。

cutover #108 — default backend 切替 里程標

目的: Rust を default backend にし scsynth を退役。前提: §2 の parity gap（A3 + A4 + γ effects）が埋まる。Done: 既定で現行 .orbs audio が Rust で SC 同等に鳴る。

OrbitStudio（VSCodium）on native（#301）cutover 後

目的: 専用アプリを native エンジンの上で。制約: scsynth は載せない・CLI + Claude 拡張が動くこと必須。並行可: engine 非依存の殻（VSCodium リブランド / node ランタイム + @julusian/midi ABI / 署名 CI）は engine 前でも着手可だが、audio が使えるのは engine 後。

β — audio DSL ⊇ pitch DSL（改良層）土台の後

目的: audio DSL を pitch DSL の上位集合に（#213 fixpitch()/time() 含む・in-process・S1 基盤）+ audio 機能（slice #239 / audio [ ] #238）。前提: C1 pitch モデルの spec 先行（pitch を audio の真部分集合として設計し、後で上位互換にできなくなるのを防ぐ）。規模: design 重め。今の .orbs DSL は表現力として充分なので急がない（やる時はそれに集中）。

6. 計測 / 耳なし検証 基盤と、その先の「聴く」ビジョン（north-star）

「人間の耳を介さずオーディオエンジンを検証する」ための計測レイヤは、本リポジトリに既に半分存在する ── verify ハーネス（#307 / #308 / #313）の解析プリミティブ（region_rms / onset 検出 / pan-from-LR / gap-RMS-floor、librosa で grounding 済）。 新トラックを発明せず、これを「計測 / 耳なし検証 基盤」として再ラベルし、その延長線上に capture と将来ビジョンを位置づける（owner の「資産再利用」志向に沿う）。

6.1 capture が load-bearing になる2つの独立 trigger

capture は単一の「cutover anchor」ではなく、性質も時期も違う2つの trigger を持つ（畳まない）:

  • (a) recovery の可聴ギャップ実測 — dog-food で「無音が痛い」と感じた時。早いかもしれないし来ないかもしれない（前倒しうる）。respawn を通る実時間出力が要るので offline では原理的に取れない。
  • (b) 耳なし実時間検証の基盤 — cutover で実エンジンに本気で乗る時、確実に必要。実時間の出力経路（scheduler→daemon→device→実サンプル）を耳でなく機械で回帰検証する。

capture seam は S1 の PostMixSink（RT 安全な post-mix tap）パターンの延長で、パターンは実証済だが production daemon には未配線。 構成 = WAV を書く sink（音声スレッド外へ ring で逃がす）+ --capture CLI + off-thread writer。A3 には畳まない（stretch 検証は offline で足りる）。

6.2 north-star:「聴く」(score-following / アルゴリズム的聴取) ── 別トラックの研究ビジョン

依存の向きと責務境界（2トラックの境界を溶かさないために） (C) = 演奏（入力音）から音響特徴を実時間抽出し、譜面（.orbs/pitch DSL 表現）+ テンポ感と照合して「今 譜面のどこか」を推定 → LLM がそれを頼りに演奏計画を立ててライブコーディング。 これは本来 WCTM（別トラック・締切 2026-08-07）側の野心で post-2.0 engine のスコープ外。 依存方向 = engine → 計測語彙 → (C)。post-2.0 engine track の責務は計測語彙をクリーンに保つことだけで、(C) のアライメント層 / 入力経路 / streaming は抱え込まない。
  • 共有してよい = 特徴抽出の「語彙」: onset / level(RMS) / spectral / pan の定義。検証でも聴取でも同じ意味を持つ純粋な計測語彙 ＝ 再利用資産。
  • 用途別に保つ = 「配管」: 検証 = batch / 自出力(既知) / アライメント不要 / 出力 tap。(C) = streaming(因果・低遅延) / 他演奏(未知) / アライメント層(DTW 等) / 入力経路。これらを見越した抽象を今 engine track で先回り実装しない（north-star は方向の記録であって先回り実装ではない＝投機的一般化の禁止）。

正直な calibration: score-following 自体は確立した MIR 研究（Antescofo / Music Plus One 等）。新規性は「OrbitScore の譜面表現に対する整合 + LLM 計画駆動のライブコーディング」という統合部分に限る（「多分誰もやっていない」はこの統合に限定して言える）。チャレンジする価値はあるが、engine track はその土台（クリーンな計測語彙）を供給するだけに留める。
