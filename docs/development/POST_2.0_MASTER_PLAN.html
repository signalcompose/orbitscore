§1. 唯一の不変条件 — permissive ライセンス規律

INVARIANT（全戦略の土台・これだけは守る） engine の依存を permissive に保ち、GPL は隔離する。 これが load-bearing なのは「機能ロック課金（freemium）は GPL では不可能」だから。permissive コアだからこそ free / freemium / クローズド / 商用を上に何でも乗せられる。
  • time-stretch = Signalsmith（permissive）。Rubber Band（GPL）は避ける。
  • Ableton Link（GPL）は別プロセス / 別 crate の隔離モジュールに留める。
  • engine は外販しない（自社内部基盤）。「engine 市場で競合する」負担を背負わない。

自コードのライセンス vs 依存のライセンス: 上の不変条件は依存（dependency）の話。自社コードのライセンスはコンポーネント別に自由に選べる（全て自社 IP）。現状は維持 — Signal compose Source-Available License v1.0（Apache 2.0 + Commons Clause + Fair Revenue $250k 閾値 + Academic 例外・source-available）。

TODO（別タスク・post-2.0 をブロックしない） Signal compose repo 間でライセンスの名称が不一致（MaxMCP=「Fair Trade License v1.0」/ godot-mcp・OrbitScore=「Source-Available License v1.0」。OrbitScore は README/docs が「Fair Trade」と呼ぶ内部不一致もあり）。理念は全 repo 共通（Apache + Commons + fair contribution + academic）。将来 canonical な名称/条項へ統一（最低限名称統一）を横断タスクで検討。2026-06-20 #ops に共有済み。

§4. 3 トラック詳細

Track A — ネイティブエンジン（Rust） DECIDED（engine=Rust） hosting=FEASIBILITY

現状 scsynth(OSC) を、既存自社 rust/ ワークスペースの発展で置換。engine=Rust は決定（既存 Rust が foundation を既にカバー + permissive が freemium を可能にする）。ただしプラグインホスティングは feasibility 確認止まりで production 実証はこれから — S1 がそれを feasibility→proof に変える。

アーキ（2026-06-21 確定・正本 POST_2.0_ENGINE_AND_DISTRIBUTION.md §2）: 薄い permissive ホスト（再生/ミックス/ルーティング/スケジュール=実装済・transport 所有）。楽器（サンプラー/audio DSL）= in-process（楽器は DSL 表現力の着地点なので flatten 境界を経由させない。in-process は表現力を自由に進化させる + 自社 Rust で隔離不要）。effects + 3rd-party = out-of-process sandboxed plugin（audio→audio の下流 / 非信頼 crash を隔離）。判定 = DSL が per-note/per-slice 制御を要する→楽器側 / 純 audio→audio→plugin 側。1st-party は standard CLAP + com.orbitscore.* 超集合拡張、shell は nih-plug。旧根拠「MIDI より豊かなため」は CLAP≠MIDI で superseded。

スコープ境界（MIDI/IAC は engine 非依存）: MIDI/IAC 出力は TS 層（interpreter → MidiScheduler → @julusian/midi → CoreMIDI/IAC）で完結する。Track A が置換するのは音声ディスパッチの OSCClient seam のみで、MIDI には触れない。唯一の接点 = クロック同期（現 TransportClock が MIDI を SC から分離して同期）→ 音声バックエンド差し替え時に「MIDI↔音声のタイミング同期」だけ再確認する（再設計ではなく統合チェックポイント）。MIDI/pitch の DSL 進化は Track C と deferred 側で扱う。

Track B — OrbitStudio アプリ（VSCodium） DECIDED（専用アプリ化） native engine の上・cutover 後

2026-06-21 確定（#302・engine-first）: OrbitStudio は native エンジンの上で動かす（scsynth は載せない）。よって audio で使い物になるのは engine cutover（#108）後（最初の /goal ではない・issue #301）。理由 = 最終的に native が乗るのに今の scsynth を Studio に同梱する工数は無駄（owner）。CLI + Claude 拡張が動くことが必須条件（VSCodium=Code-OSS が VS Code 拡張互換最良）。engine 非依存な殻（リブランド / node ランタイム + @julusian/midi ABI / 署名 CI）は再利用可で並行着手も可。VSCodium は fork ではなく上流 vscode の rebuild スクリプト群なので、段階選択できる:

  1. B1 リブランド rebuild（低保守）から始める
  2. B2 patch 付き rebuild（VST GUI 共存などの要求分だけ）
  3. B3 hard fork（Cursor 流・高保守・必要になったら）

「専用アプリにするか」ではなく「どの重さの手段で専用アプリ感を出すか」。VST GUI の Electron 共存は未確証項目（要スパイク）。配布 = Steam + Developer ID notarize の直接 .dmg（App Store はプラグインホスト型を sandbox が弾く）。notarize は既存 #210 と地続き。

Track C — Pitch モデル + Song 層（engine 非依存・並行可）

C1 pitch モデル再設計 SPEC-FIRST
pitch は直交 2 軸（anchor=root/tonic、lattice=scale/mode）× global→sequence→group カスケード。key = (tonic, デフォルトラティス)。

  • root はグループ後置に一本化: 絶対=音名 (...)F / 相対=大文字ローマ (...)bII。.root()・seq.root() は廃止（→ #280 バグはこれで消滅）。anchor は seq.key()。
  • key("C").mode("minor" | customVar)、省略で Ionian。ラティスエンジンは実装済（長ラティス + period 対応）。net-new は配線/砂糖のみ。
  • 移調は transpose()（transport は再生ヘッドと衝突するので不可）。コード名は別レイヤー（クロマティック・意図的 2 層）。
注意（spec 先行で採用） pitch モデルは specs-v2/DESIGN_DISCUSSION_RECORD.md の確定決定 #1–32 を将来変更する候補（.root() 廃止等）を含む。採用時は必ず spec を先に更新してから実装。今は決定を書き換えない。

C2 song / arrange 層 TENTATIVE — 最後・"使って検証"
転調（key over time）の置き場所であり、複数 seq を時間軸に構成する層。playable 統一（RUN/LOOP は seq でも song でも）/ part（co-active な seq 集合）/ conductor（節の基準クロック）/ 境界ポリシー（hard/musical）/ per-part key。

⚠ song 層は最もテンタティブ 大和さん曰く「記法・名前は暫定 / 使ってみないと分からない」。ハード実装に走らない。pitch（C1）と live-coding core が固まってから、実使用で検証して記法を固める。Track C 内で最後にシーケンスする。資産再利用: song 層は既存 [...]/*n/#254 section 変数の再帰適用（フラクタル・ゼロ新規ではない）。

§7. Delegation Profile — Opus / Sonnet（トークン効率）

v1.1 IMPLEMENTATION_INSTRUCTIONS.html §5 と同じ思想。判断・seam は Opus（main 直列）、固定インターフェース内の実装は Sonnet（subagent 並列）。リストに無いタスクも下記ルールで分類する。

判定ルール（これ 1 つで分類する） Sonnet: インターフェースが確定済 かつ 正しさが安価に検証可能（テスト / ビルド / spec 差分）なとき。
Opus: その作業自体がインターフェース / seam / 判断であるとき、または「誤答が安価な検証をすり抜ける」とき（← この後半が load-bearing）。
⚠ 委譲は fire-and-forget ではない — Opus が実ゲートで検証する 「agent が done と報告」≠ done。本セッションで実証済み: doc 精度の Sonnet agent は「build 緑・dead link 無し」と報告したが、実際には 5 件の実装乖離があり、さらに bot レビューが 6 件目（gate(1.2)）を発見した。→ delegated output は Opus が実ゲート（実装照合 / 実機 / spec）に照らして検証する。検証コストを織り込んで委譲する。

Opus（委譲不可・隠れ誤りが高コストな判断）:

  • A0 RT 統合設計 と S1 スパイクの RT 安全性の判定（verdict） — スパイクの価値は「驚き」。Sonnet の「動いた」は、スパイクが暴くべき RT 問題を隠す。機械的な統合サブ手順は Sonnet 可だが、結論は Opus が出す。
  • 新依存ごとの license チェック（§1） — supercolliderjs が GPL 隣接 transitive（minimatch/lodash/immutable）を誰も直接足さずに引いた件で実証済。Sonnet の crate 追加は transitive の license 違反を見逃す。
  • pitch モデルの spec 変更（Known Decisions #1–32 を動かす → spec 先行）/ song 層の記法（taste/判断）/ lexer・parser・DSL 意味論 / public API 面。

Sonnet（固定インターフェース内・検証容易 → 並列委譲）: 隔離モジュール実装（CLAP host ラッパ / time-stretch ノード / daemon コマンドハンドラ）・純関数・テスト記述・docs / 翻訳・research 要約・機械的リファクタ・spec 適合監査。いずれも「インターフェースを Opus が固定 → Sonnet が中身を埋める → Opus が実ゲートで検証」の順。
