0. なぜこのノートが要るか

engine 側のプラグインホスティング（γ out-of-process sandbox / δ VST3·AU）は POST_2.0_NEXT_STEPS.html §2・§3 にロードマップ済。 だが それを .orbs DSL からどう叩くか ── plugin 呼び出し / effect chain / send-return / aux / 出力ルーティング / automation / ファイル分割 ── は未設計。 ルーティング研究 DAW_AUDIO_ARCHITECTURE.md も 「③ DSL 表現面」を将来として留保している。本ノートはその DSL 表現面（と、それが要求する engine 要件）の設計をまとめる。

VST について: DSL から見れば CLAP も VST3·AU も「plugin」で同一。フォーマット差は engine ホスティング （γ→δ）に閉じ、DSL は format-agnostic に設計する。VST3 は SDK 3.8 で MIT 単独化済（license 障壁なし）。 δ が「最後」なのは engine 成熟度（CLAP > AU > VST3）の話で、DSL 設計上の遅延理由ではない。

1. 統合軸 ── reconciliation key = 名前

全体を貫く1本の原理 DSL は「あるべきグラフ（desired graph）」を宣言する。再評価のたびに engine は 走っている実グラフと desired を差分（diff）し、最小の delta だけ適用する （node 追加 / edge 張り替え / 削除）。変わっていない node はそのまま鳴り続ける＝音切れなし。 この差分照合（reconciliation）の key が「名前」（React の key と同じ）。 名前一致 = 同一 node →その場で再束縛（param 更新）、新規名 = 追加、消えた名 = 削除。

「mixer/aux/inst が live で増える」問題の答えがこれ。グラフが変わること自体は問題ではない ── 問題は「丸ごと teardown→rebuild すると音が切れる」こと。宣言的グラフ + 名前キー差分なら、aux を1本足すのは 「+1 node の delta」で済み、既存は無停止。「再構築でなく再束縛」がこれで成立する。

そして sum("drum") / aux("rev") / effect ハンドル / module の export 名 ── 本ノートで「名前で実体を掴む」発想が繰り返し出るのは、すべて同じ reconciliation key だから。 この1点が routing・effect ハンドル・module identity・recovery（#300） を束ねる。effect の並び替えは edge の張り替えだけで node の state（plugin 内部状態）は保持される。

2. ルーティング・グラフ

グラフは 4 種のノードと、それを繋ぐエッジで表す。エッジの向きは 「常に source が行き先を指す」（決定）。束ねる側が集めるより直感的で、変更時に認識がズレない。

ノードDSL（叩き台）トポロジーDAW 対応
sourceseq.inst("synth") / seq.audio("x.wav") / 録音入力信号の発生源（チェーン先頭）音源 / クリップ / 入力
sum（group）sum("drum")直列サブミックス（メンバの主出力がここへ）Group / Bus track
aux（send-return）send("rev") → aux("rev")並列タップ（コピーを送る・原音は継続）Return / Aux track
output（終端）output("master") / "3,4" / cue / LinkAudio ch信号の最終行き先Master / hardware out / egress

2.1 output は 2 ドメイン

ドメイン行き先備考
audiomaster / hardware ch（"3,4"）/ cue / LinkAudio channelLinkAudio egress = A4（実装済）。sum-by-name の加算は LinkAudio mixer 内で起きる egress の話で、汎用 sum とは別物。
data（MIDI）IAC bus（macOS 仮想 MIDI）音でなくデータの送り出し。seq.midi() は audio bus でなく MIDI bus へ（output() 免除・#282）。LinkAudio(音) と IAC(データ) は対称な egress。core spec L606 に「routing 予定・別 issue」。
open: output の多態と名前解決 「常に行き先を指す」を採ると output() の引数は {master / hardware ch / LinkAudio ch / cue / sum / aux} を取りうる ＝多態。"drum" が sum か LinkAudio channel か cue かの名前解決ルールを決める必要がある。 案A: output() を「次のノード」として一本化（graph edge・要名前解決）/ 案B: 終端 output() と bus 投入を別動詞にして多態を避ける。未決。

3. 信号パス / チェーン意味論

  • チェーン順 = 信号フロー。effect は順序で音が変わる（comp→eq ≠ eq→comp）ので、メソッドチェーンの並びを実行順にマップする（研究 §4「insert 順序が最重要」）。
  • source は先頭。seq は既に音源を持つ（audio() のサンプル）。inst() は音源の置換（サンプルの代わりに synth/sampler plugin）。再生でも録音でも source 先頭は不変（録音時は録る入力信号が source）。
  • play() はパターン（trigger/note）で、信号パスとは直交。
  • send() の位置 = tap 点。「effect2 の後に send を置く＝effect2 直後でタップ」と読む。DAW の固定 pre/post-fader より柔軟で、method-chain の強みを活かす。
  • mono / stereo / multi: 基本は source から伝播（推論）・必要時だけ明示。記号で everywhere に書かせると煩雑。
// 叩き台（記法は未確定）
seq.inst("synth").play(1, 0, 1, 0)   // 音源 + パターン
   .effect("comp").effect("eq")      // insert chain（この順 = 実行順）
   .send("revAux", pre)              // ← 置いた位置 = tap 点
   .output("master")                 // 終端（書かなければ master）

sum("drum").effect("glue").output("master")     // group bus 自身も chain を持つ
aux("revAux").effect("reverb").output("master") // aux も chain + output を持つ

4. エフェクトとパラメータ

  • インスタンス・ハンドルが要る。同じ effect を2回挿すと <effect名>.<param> では曖昧。ハンドルを返す形にする（叩き台: var eq1 = seq.effect("eq") → eq1.freq(...)）。これも「名前で実体を掴む」= reconciliation key。
  • effect を挿すと外に見えるパラメータが予約され、ハンドル経由で eq1.freq(800).q(1.2) のように指定 or automation 対象にできる。
  • mono/stereo は §3 の channel-count モデルに従う。

5. Automation ── 3 層

「プログラム的に param 変化を書きたい」欲は満たしつつ RT 安全（audio スレッドは毎サンプル JS を呼べない）を守るため、3 層に分ける。OrbitScore は TS 上の DSL なので automation も TS で書け、便利関数を資産化できる。

層例RT/評価便利関数で巻き取れるか
① 時間決定論ramp(0,1,4bars) / (t)=>… / for·if で書いた時間関数事前レンダして control-rate レーン / breakpoint に。RT 安全。Yes
② control-rate signal modulationLFO / envelope follower / sidechain / macro / live 入力レベル·onsetengine が mod source を primitive で持つ。DSL/TS は 「mod source → param」の配線（mod graph）を書くだけ。評価は engine が control-rate で回す。Yes（「演奏音レベルに追従」「sidechain で揺らす」はここ）
③ semantic / score-following「フレーズを聴いて」追従 / LLM 演奏計画解析 + 計画が要る。north-star（別トラック）・本スコープ外。No

線引きは「決定論 vs 反応」ではなく 「signal-rate の modulation（engine primitive・DSL は配線）」 vs 「cognitive な解析+計画」。 ①② は groove-from-structure（揺れは構造から・RNG を時間グリッドに入れない）と整合 ＝ 決定論的・構造由来に倒す。 

// 叩き台（記法は未確定）
var sweep = ramp(0, 1, 4bars)        // ① 決定論カーブ（pre-render）
filter.cutoff.automate(sweep)

var lfoMod = lfo(0.5)                 // ② mod source
synth.cutoff.mod(lfoMod, 0.3)        // mod を param に配線（control-rate 評価）
comp.thresh.sidechain("kick")        // ② 演奏音に反応（engine が follower 計算）

7. 予約キーワード = 集合 + 差分（統一イディオム）

既存の RUN(...) / LOOP(...) / MUTE(...) は 「シーケンス集合を取る予約キーワード + group-diff」（loopGroup/muteGroup を calculateLoopDiff で差分処理）。SOLO(...) はそこに soloGroup を1個足すだけ ＝同じ機構の再利用で multi-solo がタダ。 

SOLO(kick, snare, hat)   // 3つ同時 solo（集合）
SOLO()                   // 全解除

これは §1 の reconciliation（名前キー差分）と同じ思想 ── 集合も「desired な集合」を宣言し差分適用。 RUN/LOOP/MUTE/SOLO が一貫したセット演算になり、将来 BYPASS(fx...)（effect 一括バイパス）等も同型で足せる。

open: solo 意味論 soloGroup 非空 = その集合だけ可聴（solo-in-place）・他は暗黙 mute。MUTE との相互作用（solo が mute を上書きするか）と、 group/aux を solo した時（配下も鳴る）の扱いは未決。

8. Module system（ファイル分割 / import）

単一ファイルの肥大化が痛いので、トラック毎・楽器毎・play 毎に分割し、自由に import したい。reconciliation と相性が良い: 各ファイルが名前キーで node を desired graph に寄与するだけ。import = グラフの合成（名前一致で merge）。

8.1 project vs performance の分離

Global は丸ごと閉じ込めず「2 分割」 project 所有（永続・preset/variant で差し替え）: mixer / aux / instrument graph / routing / output 設定 / SR ── 再構築が音切れの元 ＝ これだけ守る。
performance 駆動（live・頻繁に書き換え・再評価が安い&歓迎）: tempo / transport start-stop / key·transpose / どの pattern を live にするか。
→ tempo は迷わず後者。「tempo を live で多用」と「project が Global を持つ」は両立する。守るのは mixer graph であって tempo ではない。 
// instruments.orbs（project 側・永続）
export var lead = global.seq.inst("synth").effect("comp").output("master")

// performance.orbs（live-coding 側）
import { lead } from "./instruments"
global.tempo(128)    // live param（再評価で再適用＝歓迎）
LOOP(lead)

hot-reload identity: performance を再評価しても import した project graph は作り直さず再束縛（§1 reconciliation）。 identity を project ランタイムに置くことで「再評価＝再束縛、再構築でない」が成立し、import 時の音切れが大半消える。 これは DAW の「プロジェクト vs クリップ」分離そのもの。

9. プロジェクト階層

clip/pattern → track → song → album。既存の deferred「song 層」（pitch model notes）の上に album（複数 song を束ねる）を載せる。これは OrbitStudio-era の project 構造。

10. Capture / render seam ── 用途は 3 つ

これまで「耳なし検証」のために deferred だった capture seam は、mixer/effects グラフと組み合わさると用途が広がる。

  1. 耳なし実時間検証（回帰テスト・#307 系）
  2. recovery の可聴ギャップ実測（#300 系）
  3. render / bounce / mastering 書き出し ← 新規価値
マスタリングも同じグラフで届く 2-track ステレオを source に → master chain（EQ / comp / limiter / normalizer） → render（capture seam） ＝ そのままマスタリング。OrbitScore が live-coding + production + mastering を1つのグラフで扱える絵になる。 capture seam が production 級（stem 書き出し / freeze / master export）の価値を持つ。

11. 決定台帳

確定（このブレストで） 決定

  • reconciliation key = 名前（宣言的グラフ + 差分適用が全体の統合軸）。
  • ルーティングの向き = 常に source が行き先を指す。
  • routing 4 ノード: source / sum(group) / aux(send-return) / output(終端)。output は 2 ドメイン（audio / data=IAC）。
  • チェーン順 = 信号フロー（source 先頭・inst()=音源置換・play()=パターン直交・send() 位置=tap 点）。
  • automation = 3 層（①時間決定論 pre-render / ②control-rate signal modulation / ③semantic=north-star 対象外）。便利関数は ①② を巻き取る。
  • Global = 2 分割（project 所有の永続グラフ / performance 駆動の live param）。
  • mixer 未挙げ項目を全部スコープ in（solo / sidechain入力 / PDC / metering / nesting）。
  • SOLO(a,b,c) = 集合ベース予約キーワード（RUN/LOOP/MUTE の group-diff 再利用・multi-solo がタダ）。
  • capture/render seam の用途 = ①検証 ②recovery ギャップ ③render/mastering。
  • VST は format-agnostic な DSL 設計で δ で差し込む（DSL 改修ゼロ）。

12. シーケンシングと関連

DSL spec 先行 → γ を de-risk γ の engine core routing graph（①）が「何を実行すべきか」は、DSL（③）が何を表現するかで決まる （チェーン topology・send/return・順序変更の意味論）。先に DSL を決めずに γ を作ると手戻りが起きる。 → 本ノートの routing-graph モデルを最初に詰めるのが最も解きやすい順序。effect chain も automation もその上に乗る。
  • A4 named channel（LinkAudio egress）= output の audio ドメイン1行目。汎用 sum とは別物（egress 固有）。
  • β audio DSL ⊇ pitch DSL: plugin/effects は audio DSL の一部。β（§3）の表現力軸と噛み合う。
  • #300 recovery: respawn 後の再 establish = 空からの full reconcile。土台は地続き。
  • #307 capture: metering / render / mastering の共有プリミティブ。
  • DAW_AUDIO_ARCHITECTURE.md: ①engine graph / ②plugin / ③DSL の3層・insert→send/return→group→master→out の層構造・PDC。
  • groove-from-structure: automation の決定論方針の根拠。
