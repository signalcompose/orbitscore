# Design Discussion Record — OrbitScore Pitch DSL

**目的**: 本記録は OrbitScore v1.1 ピッチ DSL の設計過程を、論文執筆(JASPM / ICLC 2027 / TENOR 想定)の一次資料として保存するもの。設計の最終形だけでなく、**棄却された案とその棄却理由**を残すことに主眼がある。記譜法設計における理論的選択がどのような帰結を生むかの実例として、修論の「記譜法がリズムの多様性を構造的に制約する」という主張を和声次元に拡張する素材になる。

**日付**: 2026-06-12
**参加者**: 大和比呂志(設計判断) / Claude(調査・草案・対案)
**文脈**: ICMC 2026 Hamburg 発表(Paper #721)後、ROADMAP v1.1 "MIDI Integration" の具体化。LinkAudio 統合(Epic #187)完了直後。

---

## 1. 設計の出発点

技術的動機は単純: VST ホスティング(開発コスト高)を避け、IAC Bus 経由でソフトシンセを鳴らす。しかし MIDI 出力は「音高」の概念を DSL に要求するため、ピッチ記譜の設計問題が必然的に開いた。

**根底にある問題定義(2026-06-12 の議論末尾で大和が明示)**: **「譜面を簡単に書くための DSL が存在しないこと」**。OrbitScore の自己理解として、ライブコーディング言語が結果的に譜面と同型だったのではなく、譜面記述 DSL の不在が出発点であり、ライブコーディングはその実行形態である——という因果の向きを記録しておく。既存言語の布置で言えば: LilyPond / ABC / MusicXML は譜面記述言語だが「簡単」ではなく実行(演奏)もしない。TidalCycles 系は簡単で実行可能だが記譜的構造を志向しない。**「簡単に書けて、実行可能で、記譜構造を保存する」の三点を同時に満たす空白**に OrbitScore を置く。この position claim は TENOR 投稿の骨格にそのまま使える。

初期方針(大和): クラシックの記譜ではなくバークリーメソッド的アプローチ。ただしコードシンボル(C7, Dm7)は構文に入れない。ルートからの度数を 1, 9, 3, 11, 5, 13, 7 のテンション数字で扱い、メロディーをコード構成音の度数換算で書けるようにする。

歴史的前提: v1.0 仕様(2025、アーカイブ済み)は MIDI ベースのクロマチック度数システム(1-12 = 半音階、0 = 休符)だった。v2.0/v3.0 でオーディオベースに移行した際、丸括弧は時間分割ネストに再割当てされ、v1.0 の和音記法 `(1, 5, 8)` は現行文法と衝突する状態にあった。

## 2. 設計の変遷(4 イテレーション)

### Iteration 1: グローバルハーモニーレーン(提案 → 棄却)

**提案(Claude)**: コード進行を `global.harmony` という独立タイムラインとして持ち、メロディーシーケンスは「イベント発音時刻に鳴っているコード」に対して度数解決する。`h.play()` の差し替えで全メロディーが自動リハーモナイズされる。

**棄却(大和)**: 「ハーモニーコンテキストとして和音を分けてしまうと、コードとメロディが分離してしまう」。MLTS は小節線のずれや拍子の独立を許すが、譜面の中身は通常のメロディと和声を**同時に**扱えるものであるべき。

**棄却の理論的含意(事後の分析)**: グローバルハーモニーレーンは「全シーケンスが共有する単一のハーモニックタイムライン」、すなわち**共有グリッドの再導入**だった。拍子とテンポが層ごとに独立できるのに和声だけグローバル格子に縛られるのは MLTS の思想と矛盾する。また、これはリードシートの構造(コードシンボルがメロディーとは別チャンネルとして上部に浮く)の無批判な移植であり、修論で批判対象とした記譜形式の制約をそのまま持ち込むものだった。**リードシートという記譜形式が「ハーモニーは旋律から分離した共有レイヤーである」という前提を埋め込んでいる**ことが、DSL 設計の失敗として具体的に観察された——論文での中心的エピソードになりうる。

### Iteration 2: field(root, scale) — スコープ化(提案 → 部分棄却)

**提案(Claude)**: ピッチコンテキストをリズム木のグループに付くレキシカルスコープ `.field(root, chordScale)` にする。度数はコードスケールへのインデックス(`pitch = scale[(n-1) mod 7] + oct`)。1,3,5,7 = コードトーン、9,11,13 = テンション、2,4,6 = 同ピッチクラスのオクターブ内配置が mod 7 から自然に導出される。

**この時点で確立し最終仕様まで生き残った要素**:
- ピッチコンテキストは時間的参照ではなくレキシカルスコープ(構文上の入れ子)。ライブコーディングでの予測可能性が向上。
- ハーモニックリズムが層ごとに独立 → **和声フィールドの干渉**という新しい層が追加機構ゼロで MLTS に組み込まれる(5/4 のパートと 4/4 のパートでコードチェンジの位相が周期的にずれる)。
- 転調・セカンダリードミナントが入れ子で書ける(階層的調性構造がリズム木と同じ構造原理に乗る)。
- 検証課題: 3-6-2-5 進行のサンプルで、フィールド境界をまたぐタイが「発音時ピッチ保持」の定義からコモントーンの機能再解釈(II 上の 9度 E が G7 上で 13 として読み替わる)を追加仕様なしに導出することを確認。

**部分棄却(大和)**: scale 指定を意味論の基底に置くことへの疑義。「モード自体は度数の表記で考えればいいもの」「意味論的にモードに寄せてしまうのは自由度的には下がる」。

### Iteration 3: root 基底 + mode オプトイン(確定)

**分析(Claude、大和の指摘を受けて)**: scale-index 方式は**クオリティの決定権を記譜からスケール宣言へ移す**設計だった。「同じ [1,3,5,7] がコードごとに別クオリティに解決される」一点の優雅さと引き換えに:

1. 表記の自己完結性を失う(`3` の実音が遠隔の mode 宣言に依存。譜面の一部だけ読んで歌えない)
2. 7 音コレクションに度数体系を縛る(クロマチシズムが二重写像になる)
3. **バークリーのテンション命名(T9, Tb9, T#11, Tb13)自体がルートからの音程クオリティで定義されており、スケールインデックスではない**——前案はコードスケール理論の「スケール」側に引きずられた設計であり、テンション表記の側から見れば root 方式が原型
4. LLM パイプラインで隠れ状態(遠隔文脈依存)が増える

**確定仕様**: 基底意味論は root スコープ(度数 = Ionian 基準インターバル + b/#)。E-7 のコードトーンは `[1, b3, 5, b7]` と書き、クオリティが譜面上に見える。変化記号の出現箇所 = スケール外の音、という視覚的対応が成立する(従来のコードシンボル+音名表記にない性質)。

**mode の再定義(大和)**: mode の価値は教会旋法プリセットではなく、**「度数が何を指すか」の空間そのものを演奏者が定義できる**こと。例としてアラン・ホールズワースの 2 オクターブモード。→ mode = ユーザー定義ピッチ格子(`period()` で非オクターブ周期も許容)、教会旋法はそのライブラリの一頁。v1.0 の `octmul`(度数間音程係数)の問題意識が `period()` として正統に回収された。

**構文の簡素化(大和)**: `var` 束縛 + `field()` 経由は不要。`.root(F)` をグループに直付けすればよい。var は中間オブジェクト `field()` の人工的な産物だった。

### Iteration 4: chord 値・声部タイ・合成規則(確定)

**chord 値(大和)**: `var m7 = chord([1, b3, 5, b7])` のようにルート未束縛の度数スタックを値として定義し、ライブラリ化する。「これと field でコードのルートを定義するのはちょっと別」——root はコンテキスト、chord は値、という型的分離。

**ビルダー API の棄却(大和)**: Claude が将来課題として挙げた `m7.add(9)` / `m7.omit(5)` 等に対し、「`chord([1, b3, -5, b7])` みたいに書けばいいのでは」。→ メソッド API 撤回。値合成(スタック内 spread `[m7, 9]` + 除去 `-5`)で全ケースをカバー。定義サイトと使用サイトで同一規則。`-` の照合は字面一致(解決後ピッチ照合は `-3` が文脈依存になり予測不能のため棄却)。

**声部タイ(大和の問題提起 → Claude 設計)**: 「コードワークの中でコモントーンをタイで繋ぐとかそういう処理」。→ スタック内 `_` 接頭辞(`[1, b3, _5, _b7]`)。照合は**解決後ピッチの一致**(声部位置対応ではない——声部数が異なる和音間でも壊れず、ライブ差し替え時は発音にフォールバック)。自動版 `.hold()` はスタック間限定(単音連打に適用するとリズムが消える)。ダイアトニックな 4 度進行ではコモントーンが常に「前の 1・b3 → 次の 5・7」に現れるため、voice leading の記譜がこの一機構で成立する。

**複数グループの root 共有(大和の質問 → 確定)**: 「最後の root が採用される?」に対し、タプル並置 `(...)(...).root(X)` で並置全体に適用、と確定。外側括弧で包む案は時間分割を変えてしまい、last-wins は差し替え時の消し忘れが無音事故になるため棄却。同一グループへの重複指定は diagnostic エラー。

## 3. 最終仕様の骨格(詳細は PITCH_DSL_SPEC_v1.1.md)

- `( )` 時間分割 / `[ ]` スタック(同時) / `{ }` レガート
- `_` タイ(イベントレベル)・`_n` 声部タイ(スタック内接頭辞)・`-n` 除去
- `.root(音名|キー度数)` = 基底コンテキスト、`.mode(格子)` = オプトイン、レキシカルスコープ(内→外→seq 既定)
- `chord([...])` = ルート未束縛の値、spread 合成
- 度数 = Ionian 基準 + b/#(root)/ 格子インデックス(mode)
- MIDI: IAC Bus、active note tracking、SC 経路と併走可

## 4. 論文化に向けた論点メモ

### 4.1 JASPM(記譜法がリズムの多様性を構造的に制約する)への接続

- Iteration 1 の失敗が直接の証拠: リードシート形式(コードシンボル = 分離チャンネル)を DSL に移植すると、和声が共有グリッドに縛られ、層独立のハーモニックリズムが**記述不能**になる。記譜形式の前提が表現可能な音楽の空間を切り詰める実例。
- 逆方向の証拠も: 度数 = スコープ付き解決という設計が、コードシンボル表記では書きにくい階層的調性構造(入れ子の転調)を一段の入れ子で書けるようにした。記譜形式の選択が表現空間を**拡張**する方向の実例。
- 五線譜・リードシート・DAW グリッド批判(修論前半)に対する「では代替はどう設計されるべきか」の構成的応答として位置付けられる。

### 4.2 ICLC 2027(テーマ "Dialogues")への接続

- レキシカルスコープ vs 時間的参照: ライブコーディングにおける**予測可能性**の設計原理。画面上のコード片を文脈遡行なしに音として読めるか(自己完結性)は演奏上の認知負荷に直結する。
- 和声フィールドの干渉 = MLTS の和声次元拡張。ポリメーター下でコードチェンジの位相がずれる現象は、演奏作品(ジャズコンボ + 弦楽四重奏の会場提供)で直接デモ可能。
- LLM 媒介パイプライン(Guitar → Quimera → LLM → DSL)との関係: クオリティが表記に明示される = 隠れ状態が少ない表記は、LLM による生成・解析の双方で遠隔文脈依存を減らす。「人間とLLMの双方にとって読みやすい記譜」という設計基準は Dialogues テーマに乗る。

### 4.3 TENOR(記譜テクノロジー)への接続

- mode = ユーザー定義ピッチ格子(非オクターブ周期・マイクロトーナル許容)は、度数記譜の一般化として単独で議論可能。
- タイの「発音時ピッチ保持」定義からコモントーン機能再解釈が導出される件は、記譜の意味論設計が音楽理論的現象を生成する例。

### 4.4 設計方法論として

- **不変条件の配当**: `*n` 反復とパターン変数は、ピッチ DSL の文脈(12小節 root 共有の冗長性)で生まれたが、リズム木の構造操作であるためピッチ意味論に依存せず、audio シーケンスにそのまま移転した(大和の指摘により確認)。「構造=リズム、値=コンテンツ」という設計不変条件が、一方のドメインで設計した機能を他方へ無償で移転させた最初の実例。記譜言語における直交性の価値の具体的証拠として論文で使える。逆にこの原則は `[ ]` スタックの audio 適用可否(レイヤリング)という新しい問いも開いた(spec §10-5)。
- 本記録自体が「LLM との対話による DSL 設計」の事例。大和が方向と棄却判断を、Claude が調査・形式化・帰結の分析を担う分業。ACM/Okitegami 論文(LLM コーディングエージェントの暗黙フィードバック)との接続可能性。
- 4 回のイテレーションすべてで、棄却の根拠が「理論との整合(MLTS)」「演奏時の認知負荷」「事故耐性(ライブ差し替え)」のいずれかに還元される。DSL 設計の評価軸として明文化できる。

### 4.5 リアルタイム譜面生成による人間奏者の駆動(2026-06-12 追記、別エピック)

大和の提起: ソフトシンセではなく**人間の奏者を駆動する**——弦楽四重奏 + ライブコーディング(ICMC / ICLC)。ICLC 2027 Antwerp は弦楽四重奏を会場提供しており、テーマ "Dialogues" に「コードを書く人間と譜面を読む人間の対話」として正面から接続できる。

OrbitScore がこれに構造的に向いている根拠(設計の偶然ではなく帰結):

1. **度数表記が音名スペリングを保存する**: root + Ionian 基準インターバル + b/# は綴り(D#/Eb)を一意に決定する。MIDI ノート番号派生の記譜が失う情報を、ピッチ DSL の基底意味論が最初から保持している。タイ `_`・レガート `{ }`・声部タイ `_n` も記譜要素に直接対応。→ spec §7-0(シンボリックピッチのパイプライン保持)として v1.1 実装に前提条件を埋め込み済み。
2. **quantize = 視奏ルックアヘッド**: 仕様上、`play()` 差し替えは次の小節頭まで待機するため、次小節の内容は常に1小節前に確定している。ライブコーディングの量子化機構が、そのまま初見演奏に必要な先読みバッファとして機能する。`quantize("2bar")` で猶予を延長可能。**※実装が仕様通りかは未検証(大和指摘)。v1.1 Phase 0 の検証項目に追加済み。**
3. **奏者駆動スタックの構成要素**: M5StickS3 OSC パルス配信(奏者ごとのポリリズミックメトロノーム)は**構想段階であり未実装**——譜面表示と合わせて新規開発となる。各奏者が自分の拍子・自分の小節線で譜面を見る——修論の「五線譜は MLTS を記述できない」問題への実演的応答、という位置付けは変わらない。

**経路の収束(大和の指摘による)**: 当初 Max/MaxScore 経由をプロトタイプ経路として検討したが、ネイティブ経路(node で Web を立て socket 通信で譜面を見る)が直接可能である以上、Max を経由する必要性自体がない、との指摘で棄却。Max 案の唯一の利点「追加実装ゼロ」も、Max 側セットアップを考えれば成立しない。ブラウザクライアントは奏者への配布が URL ひとつで済み、拍キュー(視覚メトロノーム、各奏者自身の拍子)も同一 WebSocket に同居できるため、初期構成は「奏者一人 = Webページ一つ」に畳み込まれる。M5StickS3 パルスデバイス構想は触覚フィードバック段階の拡張に位置付け直し。

**追加の構造的優位(2026-06-12、議論中に発見)**: play() のネスト分割木は記譜のリズム構造(連符の入れ子)と同型である。`(1, (2, 3))` はそのまま「2分割の後半が2連符」という記譜構造に対応する。MIDI からの譜面起こしに必須の転写問題(クオンタイズ・連符推定・連桁判定)が**最初から存在しない**。スペリング保存(§7-0)と合わせると、「ライブコーディング言語から譜面を生成する」のではなく**「この言語の内部表現が既にほぼ譜面である」**という主張が立つ。記譜法を批判的に再設計した言語が、結果として記譜への逆変換可能性を最大限保存していた——修論テーゼの構成的裏付けとして論文の核に近い。

なお実装の優先順位は明確で、まず v1.1 の核(ノート情報の体系的記述 + IAC → Ableton Live のソフトシンセ)。記録・エフェクトは Ableton エコシステムに委譲する、という分業がこのフェーズの設計思想。譜面エピックはその後。

先行研究系譜: INScore (Fober/GRAME、OSC 駆動リアルタイム楽譜)、SmartVox (Bell、分散譜面配信)、Decibel ScorePlayer、Quintet.net / MaxScore (Hajdu、HfMT Hamburg — ICMC 2026 開催地であり接触可能性)、Magnusson の live notation 論。

### 4.6 audio 側への波及と棄却(2026-06-12)

`[ ]` スタックの audio 適用(スライスの同時トリガー=レイヤリング)は「魅力的だが使用イメージが不明瞭」(大和)として v1.1 では開放せず、**diagnostic エラーとして構文予約**(将来の開放を純粋な追加変更にする)+ Issue 化。同時に提起された transient ベースの `slice()`(chop の等分に対するアタックポイント分割)も Issue として記録。設計リソースは「ノート情報をシンプルかつ体系的に記述できるようにする」方向に集中するという優先順位判断。

## 5. 決定ログ

| # | 決定 | 棄却した代替案 | 主な根拠 |
|---|---|---|---|
| 1 | ハーモニーはリズム木内のレキシカルスコープ | グローバル進行レーン | 共有グリッド再導入の回避(MLTS)、旋律と和声の同居 |
| 2 | 基底意味論 = root(Ionian 基準 + b/#) | scale-index 基底 | 自己文書性、クロマチシズムの自由度、テンション命名の原義、LLM 適性 |
| 3 | mode = ユーザー定義ピッチ格子(オプトイン) | 教会旋法プリミティブ | 自由度(2オクターブモード等)、octmul の正統な回収 |
| 4 | `.root()` グループ直付け、メソッド形のみ | var 必須 / プロパティ形 `.root.F` | 構文の簡素化 / `F#` で破綻 |
| 5 | chord = ルート未束縛の値、spread + `-` 合成 | ビルダー API | 一行定義で足りる言語に API は過剰、定義/使用サイトの規則統一 |

## 7. 第二議論: 記録 (2026-06-12、ピッチ DSL 確定後)

**問題提起(大和)**: ライブコーディングには時間軸の記録がない。MIDI/LinkAudio でのレコーディングは結果であり過程は消える。コードは再利用可能だが**再現性は担保されていない**。テキスト実行であることを利用し、実行ログから時間軸の記録を取り、再現を可能にしたい。

仕様は SESSION_LOG_SPEC_v1.md に確定。ここでは論文素材となる概念的決定を記録する。

### 7.1 記録/録音の区別(分析語彙として)

記録対象の三層(評価ログ/キーストローク/エンジンイベント)のうち、キーストローク・画面捕捉を大和が「それは記録ではなく録音に近い」として棄却。この線引きを概念化すると: **記録 = 原因の保存(起きたことを引き起こすもの)、録音 = 現象の保存(起きたことの像)**。評価ログは前者。日本語の「記録/録音」の対が分析語彙としてそのまま機能するため、JASPM では日本語のまま概念化する価値がある。

Seeger の prescriptive / descriptive notation(1958)との接続: 録音・スクリーンキャプチャは descriptive、楽譜は prescriptive。因果ログは **descriptive に取得され prescriptive として機能する第三項**であり、この古典的二分法を潰す。「ライブコーディングの時間軸記録」への答えが、録音でも録画でもなく**再実行可能かつ記譜可能な文書**である、という主張の理論的フック。

### 7.2 因果的同一性としての再現性

ランダム(`r`)の扱いで再現性の定義が確定した。大和: 「r=random なども**つまりは因果でしかない**ので『ランダム』という再現がされていればいい」。→ ログは原因(コード)を記録し、リプレイは再度ランダムを引く。本システムの再現性は**因果的同一性であって音響的同一性ではない**。「ここは毎回違ってよい」という演奏者の意図ごと再現される。検証層はランダム由来イベントを構造比較に落とす、という実装帰結。

### 7.3 主要な技術的決定

- **壁時計では再現できない**: quantize により、評価が小節境界の前後どちらかで反映小節が丸ごとずれる。→ 三重スタンプ(wall / transport=bar:beat / effect=解決済み効果時刻)、リプレイは音楽時間駆動。テンポ変更自体がログイベントなので参照系はログ内で自己完結。
- **イベントソーシング**: ログが真実の源泉、状態はログの畳み込み。再生モード4種: 忠実リプレイ / オフラインレンダー(faster-than-realtime) / 分岐リプレイ(`--until`、演奏のバージョン管理) / 譜面抽出(§7-0 シンボリック保持との合流——演奏終了後にその演奏の総譜が出る)。
- **分岐の出発点 = エンジン状態**(大和): エディタ復元はしない。録音層の棄却とこの決定が相互に支え合う(エディタ復元にはキーストローク層が必要だった)。
- **フライトレコーダー、起点は `global.start()`**(大和): start 前の評価(init、設定、seq 定義)は因果連鎖の一部なので、ローリングバッファからプリアンブルとして書き出す。最良の演奏は録るつもりのないセッションで起きる。
- **命名は `.orbs` から継承し同ディレクトリに並置**(大和): `<basename>.<timestamp>.orbslog`。同一ファイル複数セッションはタイムスタンプで弁別。セッションはファイルでなくエンジンに束縛され、複数ファイルセッションは per-eval の sourceFile で忠実に残る。
- **Ableton との分業の再適用**: Ableton は音響結果を、OrbitScore は因果過程を記録する。両者で演奏の完全なドキュメンテーション。

### 7.4 先行研究と新規性の線

SuperCollider History クラス(Rohrhuber / JITLib 周辺、評価コードのタイムスタンプ記録・再実行)が最直接の先例。ほか Troop(協調セッションログ)、asciinema(端末記録=「録音」側の代表)、イベントソーシング(SWE)。OrbitScore の差分: (a) quantize と音楽時間参照系の明示的扱い(SC History は壁時計ベースのはず——実装前に一次確認)、(b) シンボリック保持経由の譜面抽出、(c) 分岐リプレイ。

### 7.5 LLM バンドメンバーへの接続(次回議論、予告のみ)

大和より: Max サマースクールに提出したコンサートプロポーザル——**ライブコーディングする LLM をバンドの一員として加える**——が ICLC/ICMC に接続する。本記録仕様との関係: `.orbslog` は「どの音楽的文脈で・何を・いつ評価したか」の最小完全形であり、LLM の few-shot 素材としてそのまま機能する。**人間用とエージェント用で形式を分けない**(SESSION_LOG_SPEC 原則 5)ことで追加コストゼロ。ACM/Okitegami 論文(エージェントの暗黙フィードバック捕捉)と同じ問題系の音楽版。詳細は記録実装後に別途議論。

## 8. 第三議論: Who Conducts the Machine? (2026-06-12、記録仕様確定後)

**文脈**: Maxサマースクール・イン・藝大 2026 コンサート応募作品(2026-08-07、千住第7ホール、10分)。ドラム(吉島智仁)・トランペット(池谷恵司)・ギター(大和)の生演奏 + 自動演奏ピアノ(Disklavier 等)を、LLM(Claude)が OrbitScore のライブコーディングで駆動する。Max は参加条件(必須)であり、センシング(入口)と MIDI ルーティング+音響処理(出口)を担う。中心の問い: **学習なし・スキル(インコンテキスト)のみで自発的なバンドメンバーは成立するか**。オペレーター(森田・大石)の介助込みで音楽が成立すること自体が主題。仕様は WCTM_SYSTEM_SPEC に確定。

### 8.1 三つの時間スケールへの分業(レイテンシの設計原理化)

LLM の往復レイテンシ(数秒)を欠陥でなく設計原理に昇格: **ms = Max(センシング/ルーティング)、拍〜小節 = OrbitScore エンジン(決定論的タイミング、LOOP の自己持続)、フレーズ = LLM(意図の書き換え)**。quantize("bar") により LLM の数秒の遅延は「次の小節から効く」という音楽的に解釈可能な遅延に量子化される。プロポーザルの「ズレ・遅延・誤読を素材として扱う」のアーキテクチャ的実装。コンピングのピアニストの判断は実際フレーズ単位で起き、拍の正確さは身体(=エンジン)が担う、という音楽的対応。

### 8.2 クロック: ドラマー起点(大和の決定)

「バンドとしてはドラマーがクリックであるべき。**齟齬が面白さにつながる。そのための介助者**」(大和)。機械に人間を従わせるなら作品の問いが立たない、という作品論とアーキテクチャの一致。技術的精密化(Claude): テンポ一致だけでは小節頭の**位相**がドリフトするため、テンポ+位相の両追従が必要 = 実質 PLL。**Ableton Link を同期ファブリックとして再利用**(LinkAudio 調査済みの継承関係により tempo/beat/phase API がそのまま使える。Max 側には Link オブジェクトが存在)することで PLL 自作を回避。さらに「齟齬を面白さに」の実装として**追従の硬さ(結合度)をオペレーターの連続パラメータ化**——人間と機械の結合度そのものが演奏可能なパラメータになり、介助の仕事に音楽的次元が加わる。信頼度ゲート(トラッキング確度低下時の追従凍結)が最終防衛を兼ねる。

### 8.3 Agent Bridge = MCP サーバー(脳のない配管)

当初の「Agent Bridge」の説明が「誰が考えるか」と混線していたことを大和が指摘(「Claude Code の /monitor で Max をモニターする想定だった」)。整理: Bridge は**配管のみ**(①Max OSC→小節整列の特徴量集約、②検証つき評価投入、③ログ末尾提供)であり、**MCP サーバーとして実装**すれば「脳」= LLM ランタイムを差し替え可能になる。ランタイム A = Claude Code(即日動く、自己修復組み込み、開発ツールがバンドメンバーになるという作品的意味、画面をそのまま投影可能)、ランタイム B = 専用ループ(Messages API 直叩き、周期とレイテンシが予測可能)。**A でプロトタイピング → リハ実測で W5 に本番ランタイムを決定**。大和の Claude Code 案と MCP Bridge 案は対立せず、「モニター部分をツール化したものが Bridge」。

### 8.4 評価経路の統一が作品の問いをデータ構造に写像する

LLM・オペレーター・リプレイヤーはすべて「評価送信者」として同一経路を通る(専用経路なし)。帰結: (1) LLM の演奏は人間と同形式で .orbslog に記録され、**本番ログが次の本番の few-shot 素材になる**(ログ=エージェントの作業記憶。Okitegami の問題系の実装的回収)。(2) `evalSource: human|agent|replay` フィールドにより、**"Who Conducts the Machine?" への答えが評価単位で観測可能**——介助と自律が同一ストリーム内で識別される。作品の問いがデータ構造に写像された。

### 8.5 スキルの方針(先送り、出発点のみ確定)

スキル(LLM への楽曲枠組み)の詳細設計はイベントループ完成後で遅くない(大和)。出発点: **ジャズスタンダード(All The Things You Are 等)のリードシートを .orbs 形式にしたもの + 演奏指示書**。人間奏者のリードシートと LLM のスキルを同じ度数語彙(root 進行 + chord 値)で書く——「OrbitScore が AI と人間の理解ギャップの橋になるか」の橋の素材そのもの。リードシート .orbs はピッチ DSL(特に root(音名)、転調の多い曲での検証)の受け入れテストを兼ねる。パターン変数による語彙制約(LLM の生成信頼性装置)は後続。

### 8.6 最小実装の原則

開発スピードが勝負(本番まで約8週、SPReAD 締切 7/3 が W3 に重なる)。**作らないもの**を明示: 専用オペレーターコンソール(Max UI + Claude Code 端末 + Bridge CLI で代替)、専用投影アプリ(Claude Code 画面 + Max 表示の投影で代替)、ビートトラッカー自作(Max 既存エコシステム)、譜面表示、mode(Phase 5)、リプレイヤー(L2、本番後)。スコープ順序: **note DSL → 記録 → WCTM**(ただし L1 は Phase 1 に同乗、WCTM Bridge は評価経路確立後に並行可)。

## 9. 第四議論: pitch range / オクターブ記法 (2026-06-13、Phase 1 実機検証中)

**文脈**: Phase 1 (#228, PR #245) の MIDI 実機検証で2オクターブメジャースケールを度数 `1..15` で鳴らしたところ、大和が「ノートの度数の仕様が想定と違う」と指摘。現状 §2.1 は度数を**リニアな音階インデックス**(`1=C4 … 8=C5 … 15=C6`、式 `IONIAN[(n-1)%7] + 12·floor((n-1)/7)`)として解決しており、数字が増えるほど単純に上行する。これがバークリー的な度数語彙(メロディはルート上のコード度数で書く)と乖離していた。Iteration 3 で確定した「root 基底 + Ionian」自体は維持しつつ、**オクターブをまたぐ書き方**を再設計する議論。

### 9.1 問題の切り分け

大和の指摘は3つの実問題に分解された:

1. **`8,10,12,14` はバークリーで使わない数字**(`8`=8va はギリギリ)。リニア式は数学的に綺麗だが非音楽的な数字を露出する。
2. **メロディはルート上のコード度数で書く**: `2=9`、`4=11`、`6=13` が同ピッチクラスのオクターブ違いで対応している必要がある(現状の式でも満たされる: `2`=D4, `9`=D5)。
3. **コードトーン(1,3,5,7)のオクターブ上に名前がない**: `9/11/13` はテンション名があるが `3` の1オクターブ上は "10" とは呼ばない。現状は `3^+1`、2オクターブ上は `3^+2` だが、毎回書くと書法的に煩雑。「より直感的にメロディと和声を扱いたい」。

### 9.2 概念の分離: 和声的度数 (A) と pitch range (B)

度数記法は本来2つの独立な情報を運ぶ —— **A: 和声的度数**(コード/スケール上の位置: root, 3rd, 5th, 7th, テンション)と **B: 音域(register)**。現状はこれを1本の数字(`1..15` リニア)に融合していた。直感性のためこれを分離する:

- **A** = 音楽的な数字(`1-7` + テンション `9/11/13`)
- **B** = コンパクトなオクターブ記号(`^N`)

大和の言葉で「**pitch range(音域の状態)**」というモデルに収束。`^` は「今いる音域に入る」操作で、以降の度数はその音域で解決され、`^0` でデフォルト音域に戻る。LilyPond の relative モードに近い register state の発想。**3オクターブ上を直接書く**(`3` → E7)ケースでリニア方式との差が決定的: リニアは `24`(=3+3×7、直感ゼロ)、分離方式は **`3^3` 一発**。

### 9.3 確定した規則

- **度数 = 和声的位置**: `1-7`(root から Ionian スケール度数) + テンション `9/11/13`(メロディでも明示使用可、`2/4/6` の +1オクターブと同ピッチクラス)。変化記号 `b/#/bb/##`・detune `~` は従来通り。
- **`^N` = スティッキー pitch range**: 音または休符 `0` に付き、その地点から **running range** を base+N オクターブに設定する。`play()` 内では **読み順(時間順)に持続**し、以降のすべての度数に効く。各 `play()` の先頭で base(range 0)にリセット。`^0` で base に戻す。`0^N` = 無音で音域だけ変更(休符に付ける)。
  - **独立 `^` マーカーは無し**(裸の `^1` が play() 内に音と並ぶのは表記的に不自然、という大和の指摘)。音域だけ静かに変えるときは `0^N`。
  - 符号: `^+N` の `+` は省略可(`3^3` ≡ `3^+3`)。下方向は `^-N`。`^0` リセット。
- **range は全度数に効く(統一ルール)**: range +1 のとき `2`=D5, `9`=D6(`9` 固有の +1オクターブに range +1 が乗って合成される)。大和の根拠「スティッキーならその後に影響しないとならない」より、range は読み順で後続に作用する **linear running state**。
- **`.oct(N)` = グループのレキシカル・スコープ**(`.root()` と同じスコープ規則、持続しない)。「このまとまりだけ別音域」を安全に書く道具。**Phase 2** のグループチェーンで実装。`^N`(linear/persistent)と `.oct()`(lexical/group)は**異なる軸の道具**として併存(大和「oct() もあれば使う」)。

### 9.4 `^N`(linear) と `.root()`(lexical) の直交性 —— 実装注意

> ⚠️ **本議論では `.root()` グループ境界との相互作用を明示的には詰めていない**。確定した pitch range モデル(音域 = 時間軸の状態)と「`^N` / `.oct()` の二道具分離」決定から、spec では以下のように解決する。実装前に大和の確認を取る項目。

`.root()` は読み順上のグループに対し**レキシカル**にルートを切り替えるが、`^N` の running range は**それと独立に読み順(フラットな時間順)でスレッドされる**。両者は別の軸(root = 和声基準、range = 音域)なので、グループ境界(`.root()` のスコープ終端や入れ子タプル)は range をリセットしない。range のリセット契機は **play() の先頭** と **後続の `^M` / `^0`** のみ。入れ子タプルはリズム(時間分割)にのみ効き、running range には影響しない。グループ単位で音域を閉じたいときは `.oct(N)`(Phase 2)を使う。

**実装メモ**(Opus 向け): "スティッキー" はトークナイザの関心事ではない。パーサーは従来通り音ごとに `^N` アノテーション(octaveShift)を記録し、**play() の要素ストリームを読み順に walk するときに running range をスレッド**する。range の閉じ込め単位(フラット読み順 vs グループ構造)が §9.4 の linear/lexical 判断そのもの。度数解決の式 `pitch = rootPitch + semitones + 12·o` は不変で、`o` の供給元が「音ごとのリテラル」から「running range 状態」に変わるだけ。

### 9.5 例

```
play(1, 2, 3, 3^1, 4, 5, 1^0, 1)
      C4 D4 E4  E5   F5 G5  C4   C4
                └ ここから +1 が持続 ┘ └ ^0 で戻る
```
`3^1` で +1 音域に「入り」、以降の `4`,`5` も +1。`1^0` で base に戻る(`^` を毎回書かなくてよい —— 元の不満の解消)。

```
play(1, 5, 9, 3^2, 1)
      C4 G4 D5  E6   C6
                     └ 3^2 で range +2 になり、後続の 1 にも残る(スティッキー)
```
`9`=D5(テンション、base range)、`3^2`=E6(度数3 + range+2)、最後の `1`=C6(running range +2 が持続)。単発で1音だけ跳ばしてレンジを残したくないときは `.oct()`(Phase 2)で包む。

### 9.6 度数の受理範囲(decision #38 改訂) —— 後方互換を取らない

当初は `8,10,12,14` を「warning だが有効(後方互換)」とする方針だったが、大和が再考: 「音楽的じゃないので推奨しかねる。`14` とか描かれたくない、可読性が悪い。後方互換取る必要あるのかな、今実装してる時に。`8` はオクターブ上の1として残してもいい」。

**判断**: v1.1 ピッチ DSL は**未リリースの新機能**(PR #245 も未マージ)で、リニア高数字に依存した既存 `.orbs` は存在しない。よって**後方互換は取らず**、非音楽的な数字を**エラー**にして「オクターブは `^N` で書く」を言語レベルで一貫させる。

| 度数 | 扱い | 理由 |
|---|---|---|
| `1-7` | 受理 | スケール度数 |
| `8` | 受理 | オクターブ上のルート(`1..8` の1オクターブスケール = 8va)。`1^1` と等価 |
| `9, 11, 13` | 受理 | テンション(`2/4/6` の +8va)。メロディでも明示可 |
| `10, 12, 14` ・ `15` 以上 | **エラー** | 非音楽的・可読性が悪い。diagnostic「オクターブは `^N`(例 `3^1`)」 |
| `0` | 休符 | 従来通り |

`#11` / `b13` 等は度数 11/13 + 変化記号なので引き続き受理。2オクターブ上のテンションは `9^1` 等(度数 + range)で書く。

## 10. 第五議論: アイデンティティ・スコープ原則・表現モデル (2026-06-13、Phase 3 確定後)

Phase 3(`[ ]` スタック + chord 値)マージ後、「Pitch DSL の完成」を目標に据えるにあたり、**何を作り何を作らないか**の物差しを設計対話で固めた。トークン文法の詳細ではなく **方針(デザインプリンシパル)** を記録する位置付け(正本 PITCH_DSL_SPEC への反映は確定後)。

### 10.1 アイデンティティの言語化

OrbitScore(MIDI 側)は **「譜面的構造をプログラム的抽象化で書く DAW の MIDI 部」** である。LilyPond/MusicXML 的な **完全な楽譜再現は非目標**。これは既存設計の一貫した延長:

- 絶対音名ではなく **度数**(§2.1) / オクターブ数え上げではなく **`^N` pitch range**(§9) / ボイシングではなく **chord 値**(§6)

いずれも「リテラルな再現」より「**音楽的抽象**」を選んだ結果であり、表現記号も同じ原則で扱う。

### 10.2 スコープ判定基準(デザインプリンシパル / 非目標)

表現・記譜系の機能は、以下の **3 条件をすべて満たすものだけ採用**する:

1. **速記性**: live で素早くタイプできる(live-coding のリズムを壊さない)
2. **直交性**: 既存トークンと直交して合成できる(組合せ爆発を作らない)
3. **演奏可能性**: リアルタイムに演奏可能な MIDI パラメータに対応する

**非目標**: 楽譜の完全再現(フェルマータ/装飾音/ヘアピン/ペダル等の記譜記号の網羅)。これは将来 `@` 系や個別機能の採否を一貫した物差しで切るための原則。

### 10.3 表現の 2 軸モデル(velocity 軸 / articulation 軸)

per-note の表現で「ツリーが持つ音価」以外に必要な軸は **2 つ**に集約される:

- **velocity 軸**: per-note velocity(`@v` 系)。**アクセントはベロシティの相対ブースト**に還元(独立記号を増やさない)
- **articulation 軸**: 「割り当てスロットのうち実際に鳴る比率」。**一本の統一軸**として設計する:
  - per-seq `gate`(既存) → **per-note articulation**(staccato↔legato) → **`{ }` レガート**(sustain > slot、§5.4 / §4)

**音価(構造的な長さ)はツリー(`( )` 分割)+ タイ(`_`)が完全に持つ**。v1.0 の `@U`(絶対拍数)は **ツリー以前の競合する音価モデル**であり、復活させると「この音の長さは誰が決めるか」が二重化するため **棄却**(大和「@u は短くするだけなら他の表現もありそう」)。欠けているのは音価ではなく articulation。

**重要な含意(設計の整合)**: Phase 4 の `{ }` レガートは **すでに articulation 軸の片端**(slot より長く鳴らす)。将来の per-note articulation は同じ軸の反対端(slot より短く)。よって **articulation を一つの統一モデルとして今固める**(`{ }` を単独機構として作らない)。`@`系トークン文法の **詳細は Phase 4 後の専用フェーズ**。

### 10.4 Phase 4 スコープ確定

Phase 4(#236、§5)は **全部入り**で確定:

- `_` イベントタイ / **`_n` 声部タイ(必須 — ピアノ音源での作曲で必ず使う、大和)** / `{ }` レガート / **`.hold()` 自動コモントーンタイ(採用)**
- `{ }` は §10.3 の articulation 軸として整合設計する
